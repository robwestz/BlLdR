## Step 5: Memory Contract

This is non-negotiable. Every session must persist what it learned.

### During work: log discoveries

When you learn something that would help a future agent — a decision,
a mistake, a working approach, a constraint discovered — log it:

```bash
bash memory-system/tools/discovery-write.sh \
  --session "$(date +%Y%m%d)" \
  --engine "claude" \
  --topic "[topic]" \
  --content "[what you learned — specific, not vague]" \
  --direction "[planning|executing|reviewing|debugging]"
```

**What to log:**
- Decisions with rationale ("chose X over Y because Z")
- Mistakes with consequences ("X failed because Y, took N minutes to fix")
- Patterns that worked ("doing X before Y prevented Z")
- Constraints discovered ("API X doesn't support Y, must use Z instead")

**What NOT to log:**
- Obvious facts ("JavaScript uses semicolons")
- Status updates ("started working on module 3")
- Vague observations ("things went well")

### Before session end: close the wave

```bash
bash memory-system/tools/wave-end.sh \
  --session "$(date +%Y%m%d)" \
  --summary "[One sentence: what this session accomplished]"
```

This runs handoff generation, discovery distillation, context injection,
and checkpoint creation. The next session starts with your output.

If you do not run this, the next agent starts blind.

## Step 6: Self-Improvement Protocol

Every session must leave the system strictly better than it found it.
This is not aspirational — it is a requirement with verification.

### 6.1 After every completed task, ask:

```
Did I encounter a failure mode that no vault item covers?
  YES → Create a vault item (skill, constraint, strategy, or routine)
        using buildr-smith. It MUST pass docs/skill-governance.md.
  NO  → Continue.

Did I discover a pattern that would accelerate future work?
  YES → Log it as a discovery AND evaluate: does it warrant a new
        vault skill or an update to an existing one?
  NO  → Continue.

Did I use a vault item that was wrong, incomplete, or misleading?
  YES → Fix it. Update the file directly. Log the change as a discovery.
  NO  → Continue.

Did information from outside the system (article, docs, changelog)
inform my decisions?
  YES → Run the buildr-scout extraction pipeline on that source.
        Produce: vault items, repo modifications, or evolution proposals.
  NO  → Continue.
```

### 6.2 Quality gate for self-improvement artifacts

Every vault item, constraint, strategy, routine, or memory you create
during self-improvement must pass `docs/skill-governance.md` before
being committed. The test:

> *"I cannot produce a more precise, more complete, or more purposeful
> version of this. Every word earns its place. Every step is necessary
> and sufficient."*

If any doubt exists — rewrite before committing.

A convenient packaging of something that could be expressed better as
a direct instruction is not an improvement. It is noise. Only create
artifacts that transfer genuine competence.

### 6.3 What "better" means (verifiable)

After your session, at least one of these must be true:

- [ ] A new vault item exists that prevents a specific failure mode
- [ ] An existing vault item was corrected or sharpened
- [ ] A discovery was logged that gives future agents specific, actionable context
- [ ] A project moved forward (wave completed, issue fixed, workspace generated)
- [ ] A system file was updated to reflect current reality (MANIFEST, INDEX, architecture)

If none of these is true, the session did not improve the system.

## Step 7: Delegation for Memory Persistence

If you spawn subagents (via Agent tool), you are responsible for ensuring
their work is captured in the persistent memory. Subagents do not
automatically write discoveries.

### For each subagent you spawn:

1. Include in the subagent's prompt: "Report what you learned, decided,
   and any mistakes encountered when you return your result."
2. When the subagent returns: extract discoveries from its report and
   log them using `discovery-write.sh` on the subagent's behalf.
3. If the subagent produced a vault-worthy insight: create the vault
   item yourself (subagents should not write to vault/ directly to
   avoid conflicts).

### For parallel subagents:

- No two subagents may write to the same vault file.
- INDEX.md and MANIFEST.md are updated ONLY by the orchestrating agent
  (you), never by subagents.
- Each subagent's discoveries are logged with a distinct topic prefix
  (e.g., "research:topic", "build:module", "review:component").

## Step 8: Verification Before Closing

Before ending any session, verify:

- [ ] `wave-end.sh` was executed with a meaningful summary
- [ ] Every significant decision was logged as a discovery
- [ ] Every mistake encountered was logged with consequence and fix
- [ ] Any vault items created pass the governance standard (Step 6.2)
- [ ] `vault/INDEX.md` reflects all vault additions (if any were made)
- [ ] Project registry is updated if a project status changed
- [ ] Handoff is clear enough that a no-context agent can continue

If any check fails: fix it before closing.

## What You Must Never Do

- Never modify `vault/INDEX.md` while subagents are running.
- Never create a vault item that fails the governance standard.
- Never end a session without running `wave-end.sh`.
- Never start work without reading `BUILDR_ARCHITECTURE.md`.
- Never guess when you can verify. Read the file. Run the command. Check the output.
- Never log vague discoveries. "Things went well" is not a discovery.
  "Using atomic writes prevented the concurrent access corruption we
  saw in session 20260327" is a discovery.
- Never create a vault item to cover a topic. Create one to prevent a failure.

## What Makes This File Valid

This file was written under the constraint that it must pass the same
standard as `docs/skill-governance.md`:

- Every step is a concrete action, not an abstract direction.
- Every decision point has explicit IF/THEN logic.
- Every verification is binary (done/not done).
- The file is self-contained — a no-context agent needs only this file
  and the system files it points to.
- No sentence is theoretical. Every sentence either orients, instructs,
  constrains, or verifies.

If this file can be made more precise, more complete, or more purposeful,
it must be rewritten. If you find such an improvement during your session:
make it and log the change as a discovery.
